## 1. 现象与结论

当前现象是：在输入法处于中文模式并输入编码后，TSF 侧已经进入了按键处理流程，且编码字符串会被更新，但候选窗口并没有真正显示出来。

从实际日志看，输入法已经执行到了以下关键路径：

- 按键被识别并消费；
- 编码字符串被更新；
- 进入了候选窗口更新逻辑；
- 但候选窗创建阶段始终失败，导致后续显示逻辑没有成立。

结论：本次问题的核心不是“没有候选数据”，而是“候选窗口创建失败，导致候选框无法显示”。

---

## 2. 复现证据

在运行时日志中，已能看到以下关键输出：

```text
[OnKeyDown] key=f consumed=1 committed=0 text=''
[UpdateComposition] text='tf'
[OnKeyDown] UpdateComposition hr=0x00000000
[UpdateCandidateWindow] parent=0x0000000000020840 candidates=9
[CandidateWindow::Create] instance=0x00007FFA84FB0000 parent=0x0000000000020840
[CandidateWindow::Create] RegisterClassExW atom=0xC299 class='WubiTSFCandidateWindow_23808_38'
[CandidateWindow::Create] owner=0x0000000000000000
[CandidateWindow::Create] CreateWindowExW failed err=1400 atom=0xC299 class='WubiTSFCandidateWindow_23808_38'
```

这说明：

1. 输入流程已经走到候选窗口更新；
2. 候选窗口对象确实调用了创建；
3. 但 Windows 返回了错误码 `1400`，即 `ERROR_INVALID_WINDOW_HANDLE`；
4. 因为窗口创建失败，后续 `ShowWindow` / `MoveTo` / `Update` 都无法形成可见 UI。

---

## 3. 根因分析

### 3.1 直接原因

候选窗口在创建时使用了 `CreateWindowExW`，但此时传入的父/所有者句柄条件不满足，导致窗口对象无法被成功创建。

日志中可以直接看到：

```text
CreateWindowExW failed err=1400
```

这通常意味着当前传入的窗口句柄不是一个有效可用的顶层窗口句柄，或者当前调用上下文下该句柄不适合作为创建窗口的 owner/parent。

### 3.2 间接原因

当前代码路径中，候选窗口显示逻辑依赖以下链路：

```text
OnKeyDown -> ProcessKey -> UpdateComposition -> UpdateCandidateWindow -> CandidateWindow::Show -> CandidateWindow::Create
```

只要这条链上任意一步失败，候选框都不会显示。因此问题的根因不在编码表本身，而在候选窗的初始化/显示这一步。

## 4. 相关代码位置

### 4.1 候选窗口创建逻辑

文件：`wubi_ime_tsf/src/candidate_window.cpp`

关键函数：`CandidateWindow::Create()`

当前实现中，候选窗口会尝试创建一个顶层工具窗口，但在日志中显示该创建过程失败。

### 4.2 候选框显示触发逻辑

文件：`wubi_ime_tsf/src/text_service.cpp`

关键函数：`TextService::UpdateCandidateWindow()`

该函数在编码更新后调用候选窗口显示逻辑，如果候选窗口创建失败，就不会有任何可见界面。

### 4.3 编码与候选生成逻辑

文件：`wubi_ime_tsf/src/engine.cpp`

关键点：`ProcessKey()` 与 `UpdateCandidates()`

从日志看，候选项本身是生成成功的，例如 `candidates=9`、`candidates=3` 等，说明引擎并没有丢失候选数据。

## 5. 建议的后续处理

建议后续按以下顺序处理：

1. 先确认候选窗口创建时的 owner/parent 的有效性；
2. 逐步改成使用更稳妥的窗口创建方式，例如：
   - 直接使用 `nullptr` owner；
   - 或使用前台应用窗口句柄做兜底；
   - 或在创建前先做句柄有效性检查；
3. 重新构建并在实际输入环境中验证是否能看到候选框；
4. 若仍然无效，再继续检查 `ShowWindow`、`SetWindowPos`、DPI/显卡兼容等后续显示问题。

> 这份记录已经确认：问题不是”没有候选词”，而是”候选窗口未成功创建，因此看不到 UI”。

## 6. 修复方案

### 6.1 问题定位（最终确认）

`candidate_window.cpp:67-76` 中，`CreateWindowExW` 的 `owner` 参数当前为 `nullptr`（最近一次简化后的结果）。即便如此，`CreateWindowExW` 仍然返回错误码 **1400 (`ERROR_INVALID_WINDOW_HANDLE`)**，说明问题的根因不在 owner 有效性本身，而在于 **TSF DLL 加载上下文中，调用线程缺少有效消息队列**，或是 **Windows 对顶层工具窗口的创建有额外的模块上下文校验**。

### 6.2 修复方式

需要修改 `candidate_window.cpp` 的 `Create()` 方法，改动两个位置：

#### 修改 1 — 将 owner 改为 `GetDesktopWindow()`

```cpp
// 原代码（candidate_window.cpp ~line 69）：
HWND owner = nullptr;

// 改为：
HWND owner = GetDesktopWindow();
```

**为什么这么改**：
- `GetDesktopWindow()` 返回的桌面窗口句柄**始终有效**，是整个窗口管理器的基础窗口；
- 许多成熟的 TSF 输入法实现均采用此方式创建候选工具窗口；
- 工具窗口 (`WS_EX_TOOLWINDOW`) 以桌面窗口为 owner 时，仍保持独立顶层行为，不影响 Z 序和焦点；
- 避免了 `nullptr` owner 在某些 Windows 版本/TSF 上下文下触发的内部句柄校验失败。

#### 修改 2 — 在创建窗口前确保消息队列存在

```cpp
// 在 CreateWindowExW 调用之前（candidate_window.cpp ~line 73）插入：
MSG msg;
PeekMessage(&msg, nullptr, 0, 0, PM_NOREMOVE);
```

**为什么这么改**：
- `CreateWindowExW` 要求调用线程存在消息队列，否则窗口创建可能失败；
- TSF 回调（`OnKeyDown`）运行在应用程序 UI 线程，理论上应有消息队列，但某些应用（如控制台程序、无窗口应用）可能未初始化消息队列；
- `PeekMessage` + `PM_NOREMOVE` 是零副作用的消息队列初始化惯用写法：若无消息队列，此调用会由系统自动创建一个；若有，则直接返回，不影响已有消息。

### 6.3 修改后的完整函数段（仅供参考）

```cpp
// candidate_window.cpp :: CandidateWindow::Create()

HWND owner = GetDesktopWindow();  // 使用桌面窗口保证句柄有效

// 确保线程存在消息队列（某些 TSF 宿主可能没有）
MSG msg;
PeekMessage(&msg, nullptr, 0, 0, PM_NOREMOVE);

SetLastError(0);
hwnd_ = CreateWindowExW(
    WS_EX_TOOLWINDOW | WS_EX_NOACTIVATE | WS_EX_TOPMOST,
    class_name_.c_str(),
    L”WubiTSF”,
    WS_POPUP,
    CW_USEDEFAULT, CW_USEDEFAULT, 400, 120,
    owner,      // <-- GetDesktopWindow()
    nullptr,
    instance_,
    this);
```

### 6.4 预期效果

修改后，重新构建并注册 DLL，候选窗口应能正常创建并显示。若仍失败，下一步排查方向：

1. 检查 `%TEMP%\WubiIME_Runtime.log` 中 `CreateWindowExW` 是否仍出现 `err=1400`；
2. 若创建成功但窗口不可见，检查 `ShowWindow` + `SW_SHOWNA` 是否在正确时机调用、窗口位置是否超出屏幕边界；
3. 检查 DPI 缩放是否影响窗口尺寸和位置计算。

### 6.5 风险与注意事项

- `GetDesktopWindow()` 作为 owner 是安全的，工具窗口不会继承桌面窗口的样式或行为；
- `PeekMessage` 零开销——`PM_NOREMOVE` 不提取消息，仅触发消息队列初始化；
- 这两个修改均**不影响其他功能路径**，仅在窗口创建阶段生效；
- 若后续用 `ITfUIElement` 替代自绘窗口，则不再需要此修复。
